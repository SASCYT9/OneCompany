# One AI V2: план каталожного інтелекту

Дата: 17.07.2026  
Статус: архітектурний план, без застосування міграцій до production

## 1. Ціль

One AI має шукати не по окремих брендах або випадкових ключових словах, а по всьому єдиному каталогу One Company:

- знати кожен опублікований товар і кожен його варіант;
- читати назви, описи UA/EN, теги, характеристики, метаполя та дані сумісності;
- розуміти природні запити, скорочення й помилки;
- повертати тільки товари, які пройшли жорсткі умови запиту;
- не вигадувати сумісність, ціну, наявність або характеристики;
- чесно ставити уточнювальне питання чи повертати `no_match`, якщо доказів недостатньо;
- покращуватися на перевірених виправленнях і результатах використання без самовільної зміни даних про сумісність.

## 2. Що є зараз

### Критичні знахідки

1. У Prisma та міграції описана `ShopProductKnowledge`, але в поточній підключеній базі цієї таблиці немає. Спроба синхронізації повернула Prisma `P2021`.
2. Отже, поточний One AI фактично не має заповненого індексу знань по всьому каталогу.
3. Поточний semantic ranking не знаходить нові товари. Він лише переставляє вже знайдені кандидати і працює тільки з `SHOP_AI_SEMANTIC_RERANK=1`.
4. Поточний маршрут спочатку викликає загальний HTTP-пошук каталогу, потім залишає до шести результатів. Це не гарантує повноту вибірки.
5. `applications` зберігаються у JSON. Для точного пошуку за пов'язаною комбінацією «марка + модель + кузов + роки + двигун + ринок» потрібні окремі індексовані рядки сумісності.
6. Зараз немає автоматичного процесу, який після кожної зміни товару гарантовано перебудовує knowledge record та embedding.
7. Немає моделі зворотного зв'язку One AI й контрольованого процесу, який перетворює підтверджені виправлення на покращення даних або правил.
8. Поточний eval-набір містить лише вісім запитів і переважно перевіряє розбір фрази, а не повноту та безпечність виданих товарів.
9. Поточний storefront search допускає make/model/chassis через збіг у загальному тексті, а не обов'язково в одному application. При нульовому результаті він також має fallback, що послаблює vehicle-фільтри. Для звичайного каталогу це може бути корисним, для суворої відповіді One AI — небезпечно.
10. Поточний candidate gate пропускає не тільки підтверджену сумісність, а й статус `unknown`; строгий режим повинен приймати лише доказаний збіг.

### Поточний data-quality snapshot

Цифри нижче належать локальним snapshot/audit-файлам і мають бути повторно виміряні безпосередньо з development/staging DB перед міграцією:

- у каталожному cache: 14 992 товари та 12 655 варіантів;
- 14 386 товарів мають long description;
- 4 339 товарів, або 28,9%, не мають англомовного опису;
- fitment audit охопив 14 934 товари;
- make визначено для 92,5%, model — 58,4%, chassis — 32,2%, year — 72,6%;
- 5 129 товарів, або 34,3%, мають автоматичний статус `needs_review`.

Тому «проіндексувати 100% товарів» і «дозволити 100% товарів для суворого підбору» — різні речі. Індексувати потрібно весь каталог; рекомендувати як точну сумісність можна тільки підтверджену частину. Решта має бути видимою в адмін-черзі як робота з якістю даних.

## 3. Що означає «навчатися на нашій базі»

Не потрібно регулярно донавчати мовну модель на мінливому каталозі. Для каталогу правильна модель — актуальна база знань плюс retrieval:

1. Каталог є єдиним джерелом правди.
2. Окремий індексатор нормалізує всі поля, читає описи та створює пошукові представлення.
3. При кожному запиті One AI отримує актуальні факти з локальної бази.
4. Підтверджені менеджером виправлення покращують нормалізовані дані, словник і тести.
5. Кліки та покупки можуть впливати лише на м'який порядок релевантних товарів, але ніколи не скасовують правила сумісності.

Це дає оновлення знань одразу після зміни каталогу, зрозуміле походження кожного факту і можливість суворо відсіювати несумісні товари.

## 4. Цільова модель даних

### `ShopProductKnowledge`

Один агрегований запис на товар:

- нормалізована назва, бренд, категорія та тип деталі;
- пошуковий текст UA/EN;
- загальні перевірені характеристики;
- версія схеми, `contentHash`, статус індексації;
- статус якості даних: `verified`, `automatic`, `needs_review`, `blocked`.

### `ShopProductApplication`

Один рядок на одну пов'язану автомобільну або мото-сумісність:

- `productId`, опційно `variantId`;
- тип транспорту;
- марка, сімейство моделі, модель;
- кузов/шасі;
- `yearFrom`, `yearTo`;
- код/сімейство двигуна;
- тип кузова, привід, трансмісія;
- ринок;
- `opfGpf`: `with`, `without`, `both`, `not_applicable`, `unknown`;
- джерело факту, confidence, статус перевірки;
- посилання на поле/фрагмент опису, з якого взято факт.

Ці поля мають бути індексованими. Жорсткий фільтр виконується по одному application-рядку, щоб марка з одного опису не могла помилково поєднатися з роком або двигуном з іншого.

### `ShopVariantKnowledge`

Один рядок на варіант:

- SKU;
- option values;
- матеріал, діаметр, сторона, колір, тип насадок та інші variant-level факти;
- власна сумісність, якщо вона відрізняється від товару;
- зв'язок з актуальною ціною та залишком у головних таблицях.

### `ShopKnowledgeChunk`

Окремі невеликі фрагменти змісту:

- `productId`, опційно `variantId`;
- locale;
- source field: title, short description, long description, highlight, metafield, manual;
- section/title;
- очищений текст;
- `contentHash`;
- embedding і модель embedding.

Embedding потрібен для змісту на кшталт «хочу гучніший звук без дрону», але не є доказом сумісності.

### Категорійні атрибути

Для різних правил 15 брендів потрібен реєстр атрибутів:

- `ShopAttributeDefinition`: ключ, тип, одиниця, категорія, допустимі значення, чи є умова жорсткою;
- `ShopProductAttributeValue`: товар/варіант, текстове, числове або булеве значення.

Так зберігаються спеціальні фільтри магазинів без втрати унікальних полів, але загальний каталог отримує одну узгоджену модель.

### Навчання та аудит

- `ShopAiQuery`: оригінальний запит, нормалізовані умови, версії parser/retriever/validator, час і вартість.
- `ShopAiCandidateDecision`: accepted/rejected та машинний reason code для кожного кандидата.
- `ShopAiFeedback`: `relevant`, `irrelevant`, `wrong_fitment`, `wrong_category`, `no_result`, `manager_corrected`.
- `ShopAiEvaluationRun`: версія системи, набір тестів, метрики, регресії.

### `ShopKnowledgeOutbox`

Черга переіндексації:

- `productId`, reason, target content hash;
- attempts, `availableAt`, `lockedAt`, `lastError`;
- dead-letter/retry state;
- desired index schema/model version.

У проєкті багато різних шляхів запису товарів та імпортерів, тому надійніше ставити outbox event на рівні БД для змін `ShopProduct`, variant, option, metafield і fitment, а worker робити ідемпотентним. Це не дозволить одному забутому importer-у залишити One AI зі старими знаннями.

## 5. Індексація всього каталогу

### Початковий backfill

1. Застосувати міграції спочатку у development/staging.
2. Пройти всі опубліковані товари, а не одну сторінку і не один бренд.
3. Зчитати товар, варіанти, options, metafields, tags, collections, UA/EN описи та існуючий fitment.
4. Нормалізувати category, product kind, vehicle applications і атрибути.
5. Створити chunks з усіх змістовних полів.
6. Побудувати embedding лише для нових або змінених chunks.
7. Винести неоднозначні дані в чергу `needs_review`, не робити їх підтвердженою сумісністю.
8. Звірити counts і заборонити запуск strict режиму, доки coverage gate не пройдено.

### Оновлення після змін

Після імпорту або редагування товару створюється idempotent knowledge job:

`product_saved -> contentHash -> normalize -> validate -> upsert applications/attributes/chunks -> embed changed chunks -> coverage status`

Не можна покладатися на ручний запуск script. Потрібні outbox/queue або гарантований виклик індексатора з усіх шляхів імпорту. Повторний запуск з тим самим hash нічого не переобчислює.

Товар переходить у `ready` атомарно лише після побудови всіх обов'язкових chunks, structured facts і embeddings. Між оновленням тексту та завершенням embedding не можна видавати змішану стару/нову версію знань.

### Правила extraction

1. Спочатку структуровані джерела: metafields, tags, variants, options, перевірений fitment.
2. Потім детерміновані parsers.
3. LLM може запропонувати JSON за суворою схемою лише для неструктурованих описів.
4. Кожен LLM-факт повинен мати джерело й доказ у тексті.
5. Непідтверджена або суперечлива сумісність має статус `needs_review` і не проходить strict validator.

## 6. Суворий pipeline запиту

```text
User message
  -> language/typo/synonym normalization
  -> intent + vehicle + hard/soft constraints
  -> ambiguity check
  -> indexed SQL eligibility filter
  -> lexical + semantic retrieval inside eligible set
  -> deterministic compatibility validator
  -> rank only accepted products
  -> hydrate live price/stock/publication
  -> grounded response with reason codes
```

### Крок 1. Розбір запиту

Виділяємо:

- намір і категорію;
- тип деталі;
- марку, модель, кузов, рік, двигун, тип транспорту;
- OPF/GPF, ринок, кузов, трансмісію;
- бренд, матеріал, діаметр, потужність, бюджет;
- що є жорсткою умовою, а що побажанням.

Детермінований parser обробляє відомі моделі, скорочення, роки, SKU й типові помилки. LLM потрібна тільки для неоднозначного природного формулювання і повертає schema-validated JSON.

### Крок 2. Уточнення

Якщо одна відсутня відповідь може змінити сумісність, One AI ставить коротке питання до пошуку. Наприклад: кузов, двигун або OPF/GPF. Не можна мовчки послаблювати умову.

### Крок 3. Eligibility

SQL вибирає повний набір допустимих product/variant IDs за пов'язаними application-рядками й жорсткими атрибутами:

- `isPublished = true`;
- правильний scope;
- правильна категорія і product kind;
- точна/допустима марка, модель, кузов;
- рік входить у діапазон;
- двигун/ринок/OPF збігаються, якщо задані;
- variant-level умови збігаються;
- бюджет і доступність застосовуються за правилами запиту.

До цього етапу не можна ставити `LIMIT 6`, інакше частина правильних товарів ніколи не потрапить до валідатора.

### Крок 4. Hybrid retrieval

Всередині допустимого набору об'єднуються:

- exact SKU/brand/model matches;
- PostgreSQL full-text/trigram пошук;
- semantic search по chunks;
- category-specific attributes.

Semantic score ніколи не повертає відхилений жорсткими правилами товар.

### Крок 5. Validator

Кожен кандидат отримує статус:

- `exact`: усі задані hard constraints підтверджені;
- `needs_clarification`: немає достатнього факту для безпечного рішення;
- `rejected`: є конкретна суперечність.

Для відхилень зберігаються reason codes: `WRONG_CATEGORY`, `WRONG_CHASSIS`, `YEAR_OUT_OF_RANGE`, `ENGINE_MISMATCH`, `OPF_MISMATCH`, `VARIANT_MISMATCH`, `UNVERIFIED_FITMENT` тощо.

У головній видачі показуються тільки `exact`. `needs_clarification` використовується для питання користувачу, а не маскується як рекомендація.

### Крок 6. Відповідь

LLM не отримує право називати довільні товари. Їй передається готовий список accepted IDs і дозволені факти. Ще надійніший базовий режим — шаблонна відповідь без другого LLM-виклику.

Ціна, валюта, публікація та залишок беруться з живих таблиць при формуванні відповіді, а не з embedding.

### No-match

Якщо точних результатів немає:

- сказати, яка умова не дала підтвердженого збігу;
- попросити одну потрібну характеристику;
- запропонувати передати VIN/контекст менеджеру;
- не підміняти результат товарами для іншого кузова, року або типу деталі.

## 7. Приклад: «вихлоп на M3 2018»

1. Intent: product search.
2. Category: exhaust.
3. Product kind: повна система, якщо користувач не попросив downpipe/link pipe/tips.
4. Vehicle: BMW M3, year 2018.
5. Vehicle resolver визначає допустимий chassis на основі перевіреної таблиці поколінь.
6. Якщо для цього року/ринку в каталозі є взаємовиключні OPF-варіанти, бот уточнює OPF/ринок або VIN.
7. SQL шукає тільки exhaust systems з application, де BMW + M3 + chassis + 2018 належать одному рядку.
8. Validator відкидає G80/E92, чіп-тюнінг, окремі насадки, несумісний OPF і товари з непідтвердженою сумісністю.
9. Semantic ranking сортує лише підтверджені системи за побажаннями «гучніше», «титан», «без дрону» тощо.

## 8. Контрольоване навчання

### Що може покращувати систему

- підтверджена менеджером правильна/неправильна сумісність;
- виправлення категорії або product kind;
- доданий alias моделі, кузова, двигуна чи типу деталі;
- запити без результату, для яких у каталозі насправді є товар;
- релевантні кліки, додавання в кошик і замовлення як soft-ranking signals.

### Що не може автоматично змінювати hard rules

- одиничний клік;
- популярність товару;
- текст, згенерований LLM;
- непідтверджене повідомлення користувача;
- semantic similarity.

Менеджерські виправлення проходять review, після чого перебудовують knowledge record і стають regression-тестом. Так One AI навчається без накопичення небезпечних помилок.

## 9. Тести та метрики приймання

### Coverage gate

- 100% опублікованих товарів мають актуальний knowledge record;
- 100% опублікованих варіантів проіндексовані;
- 100% непорожніх UA/EN описів розбиті на chunks;
- 0 knowledge records зі старою schema version перед strict rollout;
- окремо відомий відсоток товарів з verified fitment і `needs_review`.

Товар без verified fitment може бути знайдений у нефітментному запиті, але не повинен заявлятися сумісним з конкретним авто.

### Golden set

Потрібно щонайменше 500 перевірених кейсів:

- усі категорії та всі 15 брендів;
- UA, EN, суржик, трансліт, скорочення й помилки;
- короткі й багатокрокові діалоги;
- суміжні покоління, граничні роки, різні двигуни;
- OPF/non-OPF;
- запити на конкретний SKU/variant;
- свідомо відсутні товари;
- негативні assertion: які SKU категорично не можна показати.

### Release gates

- 0 несумісних товарів у golden hard-constraint set;
- 0 товарів іншої категорії/product kind;
- 100% facts у відповіді мають provenance у БД;
- recall усіх відомих допустимих SKU >= 98%;
- no-match не розширює silently hard constraints;
- stale index P95 < 5 хвилин;
- availability системи не залежить від LLM provider;
- P95 deterministic retrieval < 1.5 с, повна відповідь < 3 с;
- вартість embedding виникає лише для зміненого контенту.

## 10. Спостережуваність

Для кожного запиту потрібна приватна trace:

- parsed constraints;
- normalized vehicle;
- кількість кандидатів після кожного етапу;
- accepted/rejected IDs і reason codes;
- версії index/parser/validator/model;
- latency кожного етапу;
- кількість provider calls і токенів;
- clarification/no-match/manager handoff;
- click/cart/order outcome без збереження зайвих персональних даних.

Панель якості повинна показувати coverage, stale/failed jobs, no-result rate, wrong-result feedback, найчастіші невідомі aliases і regression status.

## 11. Етапи реалізації

### Етап 0 — відновити фундамент (2–3 дні)

- перевірити migration state у development/staging;
- застосувати knowledge migration не в production;
- створити coverage audit і dry-run;
- зафіксувати baseline каталогу й eval.

Готово, коли counts узгоджені й індекс можна безпечно перебудувати повторно.

### Етап 1 — Knowledge Schema V2 (4–6 днів)

- relational applications;
- variant knowledge;
- chunks;
- category attribute registry;
- provenance, confidence, review status;
- індекси й constraints.

### Етап 2 — повний ingestion (5–7 днів)

- adapters для всіх 15 брендів;
- full backfill;
- incremental/outbox jobs;
- content-hash deduplication;
- extraction review queue;
- coverage dashboard.

### Етап 3 — Strict Retrieval V2 (5–7 днів)

- query normalization;
- constraint compiler;
- direct Prisma repository без внутрішнього HTTP;
- eligibility SQL;
- hybrid lexical/vector retrieval;
- deterministic validator і reason codes;
- exact/no-match response contract.

### Етап 4 — діалог, feedback та eval (4–6 днів)

- typed conversation state;
- feedback capture;
- manager correction workflow;
- 500+ golden cases;
- database-backed coverage/recall eval;
- cost/latency telemetry.

### Етап 5 — безпечний запуск (5–7 днів)

1. Shadow mode: V1 відповідає, V2 лише логуються.
2. Порівняння accepted/rejected та ручна перевірка розбіжностей.
3. Внутрішній canary.
4. 10% користувачів із kill switch.
5. 50%, потім 100% тільки після release gates.

Орієнтир: 4–6 тижнів для якісної версії, що охоплює каталог, а не один демонстраційний запит.

## 12. Перший implementation slice

Перший завершений вертикальний зріз має бути не «ще один кращий prompt», а:

1. migration/coverage audit;
2. `ShopProductApplication` + `ShopKnowledgeChunk`;
3. idempotent backfill усіх опублікованих товарів;
4. strict repository з direct Prisma;
5. один end-to-end сценарій по exhaust, але з усіма товарами цієї категорії;
6. automated assertions, що кожен відомий сумісний SKU знайдено, а кожен несумісний відхилено;
7. після доказу архітектури — поширення adapters на інші категорії та бренди.

Це створює фундамент, на якому One AI може безпечно «знати» весь каталог, а не накопичувати винятки для окремих фраз.
