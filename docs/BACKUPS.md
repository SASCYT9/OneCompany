# Бекапи бази даних One Company SHOP

## Де живе база даних

- **Продакшн:** PostgreSQL (Vercel Postgres / Prisma), підключення через змінну середовища `POSTGRES_URL` або `DATABASE_URL`.
- На хостингу Vercel **немає доступу до `pg_dump`** у рантаймі, тому кнопка «Зробити бекап зараз» в адмінці на проді **не працює**. Вона корисна лише в середовищах, де є shell і встановлений PostgreSQL client (наприклад, локально або на VPS).

## Як робити бекапи на проді

Рекомендований спосіб — **GitHub Actions** workflow у репозиторії:

- **Файл:** `.github/workflows/backup-database.yml`
- **Запуск:** щодня о 03:00 UTC (cron) або вручну: Actions → Backup database → Run workflow.
- **Секрет:** у GitHub Settings → Secrets додайте `DATABASE_URL` (рядок підключення до PostgreSQL; той самий, що для деплою).
- **Результат:** дамп завантажується як артефакт збірки з retention 7 днів. Скачати: збірка → Artifacts → postgres-backup-N.

Для довгострокового зберігання (наприклад, 7 денних + 4 тижневі) додайте у workflow крок завантаження в S3, R2 або інше сховище.

## Політика зберігання (рекомендація)

- **Щоденні бекапи:** зберігати останні 7 днів.
- **Тижневі бекапи:** зберігати останні 4 тижні.
- Після цього старі файли можна видаляти вручну або скриптом, щоб не переповнювати сховище.

## Сторінка «Бекапи» в адмінці

- **Оновити список** — показує файли з папки `backups/` на сервері (якщо така є і API має до неї доступ). На Vercel ця папка ефемерна і не зберігається між деплоями, тому список на проді зазвичай порожній.
- **Зробити бекап зараз** — викликає API, який намагається виконати `pg_dump`. На Vercel це завершиться помилкою (немає `pg_dump`). Використовуйте бекапи через GitHub Actions або ручний дамп з машини, що має доступ до БД.

## Відновлення з бекапу

- Бекапи — це SQL-дампи (або custom/archive формат `pg_dump`).
- Відновлення: через `psql` (plain SQL) або `pg_restore` (custom/directory archive).
- Перед відновленням варто зупинити застосунок або перевести його в режим обслуговування і зробити поточний дамп стану БД.

Приклад (plain SQL):

```bash
psql "$POSTGRES_URL" < backup.sql
```

Для production завжди узгоджуйте відновлення з DevOps і перевірте резервну копію перед виконанням.
